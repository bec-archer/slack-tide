Overview

What is QRSTKR?

QRSTKR is a QR code sticker platform for tracking physical items — trucks, boats, equipment, you name it. Customers buy a custom QR sticker shaped like a US state, stick it on their thing, and that sticker links to a digital profile with specs, photos, maintenance records, and ownership history. When the item gets sold, the ownership transfers but the sticker stays. Repair shops can scan and log service records too.

The business loop: someone sees a sticker → scans it → thinks "that's cool" → buys their own → the flywheel spins.

Infrastructure

Tech Stack & Architecture

Frontend

Next.js 16 with React 19, TypeScript, and Tailwind CSS. The app uses server-side rendering and API routes built into Next.js — no separate backend server.

Hosting & Deployment

Vercel — auto-deploys from GitHub (github.com/bec-archer/qrstkr). Custom domain qrstkr.com is on Cloudflare DNS.

Database, Auth & Storage

Supabase is the central hub — it handles three things:

  • PostgreSQL Database — all application data (stickers, items, users, orders, maintenance records, ownership history)
  • Supabase Auth — email/password plus OAuth (Google and Apple). Session management via middleware that refreshes tokens on every request.
  • Supabase Storage — photo uploads, QR code images, and file attachments. Client-side compression happens before upload (Canvas API, 1200px max, 80% JPEG quality). Photo metadata is stored as JSONB on the items table.

QR Code Generation

Railway — a separate Python microservice (containerized via Docker, ~$5/mo) that generates the state-shaped QR codes. Supabase triggers generation when a new order comes in, and the generated QR image is saved back to Supabase Storage.

Payments

Square — handles all payment processing. Payment info goes directly from the user's browser to Square's servers — it never touches the QRSTKR backend. Square confirms payment → order record created in Supabase.

AI Assistant (Planned)

Claude API (Anthropic) — will power an AI assistant on item pages that can answer questions about equipment using the item's data and maintenance history as context.

Email/Notifications

An email service (TBD) for transfer requests, renewal reminders, and shop submission notifications.

Security

Authentication Details

The auth system uses Supabase Auth with three providers:

  • Email + Password — standard signup/login
  • Google OAuth — "Continue with Google"
  • Apple OAuth — "Continue with Apple" (JWT secret expires every 6 months — next rotation ~August 2026)

Session persistence is handled by Next.js middleware (src/middleware.ts) that creates a Supabase server client on every request and refreshes the session token. The AuthContext (src/contexts/AuthContext.tsx) provides a useAuth() hook for client components.

Auth redirects are handled via URL params (e.g., /auth?redirect=/claim/XYZ).

Security

Row Level Security (RLS)

Supabase RLS policies are already in place:

  • Public read — anyone can view item profiles (that's the whole point of scanning)
  • Owner update — only the item owner can edit their item details
  • Anyone can claim/insert — unclaimed stickers can be claimed by any authenticated user
  • Profiles — users can only edit their own profile

Your Domain

Cybersecurity Task List

These are the security-relevant tasks extracted from the project roadmap, organized by priority.

Critical / Immediate

RLS Policy Audit & Hardening

The existing RLS policies cover the basics (public read, owner write) but haven't been stress-tested. Audit every table for edge cases — especially around the orders, order_items, and coupons tables that are about to go live. The migration SQL is at qrstkr/supabase/migrations/20260303_orders_and_coupons.sql and needs to be reviewed before it's run.

API Route Security Review

The Next.js API routes handle order creation, admin operations, and coupon validation. Review these for:

  • POST /api/orders and GET /api/orders — proper auth checks
  • PATCH /api/orders/[orderId] — only admins or the order owner should be able to update
  • GET/PATCH /api/admin/orders — admin access is currently checked via email. Needs a proper admin role system.
  • POST /api/coupons/validate — rate limiting to prevent brute-force coupon guessing

Square Payment Flow Security

Payment goes browser → Square directly (good — PCI compliance is on Square). But verify that the webhook/callback from Square to Supabase is authenticated and can't be spoofed. Order confirmation should be validated server-side, not trusted from the client.

Apple OAuth Secret Rotation

Apple JWT secret expires every 6 months. Next rotation is ~August 2026. Set up monitoring/alerting so this doesn't silently break login. Document the rotation procedure.

High Priority

Short Code Security

The allocate_short_code database function generates codes for sticker URLs (qrstkr.com/i/XXXXXXXX). Review the character set (ambiguous chars already excluded) and ensure codes aren't sequential or predictable — someone shouldn't be able to enumerate valid sticker URLs.

Photo Upload Validation

Client-side compression is done before upload, but there's no server-side validation mentioned. Implement:

  • File type verification (not just extension — check magic bytes)
  • Max file size enforcement server-side
  • Image dimension limits
  • Malicious file detection (SVG XSS, polyglot files)
  • Supabase Storage bucket permissions review

Ownership Transfer Security (Upcoming)

The transfer flow involves generating invite links. When this gets built:

  • Transfer tokens must be single-use, time-limited, and cryptographically random
  • The old owner must explicitly confirm
  • The new owner's acceptance must be authenticated
  • Ownership history must be immutable (append-only)

Shop System Authentication (Upcoming)

Repair shops will be able to write maintenance records to items they don't own. This is a significant trust boundary:

  • Shop verification/approval process before they can submit
  • Rate limiting on shop submissions
  • Owner dispute/delete flow needs audit trail
  • Prevent shops from modifying existing records (append-only)
Medium Priority

Rate Limiting & Abuse Prevention

  • QR scan endpoint — prevent scraping of all item profiles
  • Auth endpoints — brute force protection on login
  • Coupon validation — prevent code enumeration
  • API routes — general rate limiting per IP/user

Claude API Integration Security (Upcoming)

When the AI assistant is added:

  • API key management (never exposed client-side)
  • Per-user rate limiting to control costs
  • Input sanitization (prevent prompt injection via item data)
  • Conversation history access control (owner-only)

Code Obfuscation Review

The sticker customizer uses a Babel → javascript-obfuscator pipeline for production. Review the obfuscation config to ensure no sensitive logic or API keys are leaking through the client-side code.

HTTPS & Transport Security

All traffic is HTTPS (good), but verify:

  • HSTS headers are set on Vercel
  • No mixed content issues
  • Supabase Storage URLs are served over HTTPS
  • Railway service communication is encrypted
Lower Priority / Ongoing

Marketplace Security (Phase 3)

When the marketplace launches: inquiry/messaging system needs XSS prevention, spam filtering, and content moderation. Listing data needs validation to prevent injection.

Dealer Portal (Phase 4)

Bulk operations = bigger blast radius. Role-based access control, audit logging, and API key scoping will all be needed.

Native Mobile App Security (Phase 4)

Certificate pinning, secure token storage, biometric auth integration.

Subscription/Renewal System (Phase 4)

Recurring payment security, grace period handling, sticker deactivation flow.

Context

Key Architecture Decisions

Security-relevant design choices baked into the platform.

  • No self-hosted servers. Everything is managed cloud services (Vercel, Supabase, Railway, Square). This reduces attack surface but means security depends heavily on proper configuration of each service.
  • Supabase is the brain. Nearly everything connects through it. If Supabase RLS is wrong, the whole thing is wrong.
  • Payment never touches us. Square handles PCI compliance. We only get confirmation callbacks.
  • The sticker URL is public by design. qrstkr.com/i/XXXXXXXX is meant to be scannable by anyone. The security boundary is between "viewing" (public) and "editing" (authenticated owner only).
  • Client-side image processing. Compression happens in the browser before upload. Server-side validation is still needed as a second line of defense.
